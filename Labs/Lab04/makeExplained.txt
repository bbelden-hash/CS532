suppose you have ->
gettime.h
gettime.c
lab04.c

when you compile manually, you could go
gcc -Wall -g -c gettime.c
gcc -Wall -g -c lab04.c 
gcc -Wall -g -o lab04 gettime.o lab04.o
this works

but imagine you only changed one line in lab04.c
do we really want to compile everything again?
no

gettime.c hasnt changed 

a Makefile lets the "make" program figure out
"only recompile the files that changed"
saves time, especially in large projects


creating a variable, CC stands for C Compiler
CC = gcc

so whenever Make sees --> $(CC) it replaces it with gcc

CC -> gcc

another variable
CFLAGS = -Wall -g

"CFLAGS" means compiler flags
these options tell GCC how to compile -->
-Wall
turns on lots of useful warning messages ... GCC warns you if you forget to initialize a variable, etc.
addition of debugging information
-g
this lets programs like "gdb" debug your code
when youre finished debugging you often replace it with --> -O, optimizes your program to run faster

$(CFLAGS) -> -Wall -g

dependencies
DEPS = gettime.h Makefile

says, "my source files depend on these files."
if you change "double gettime(void);" inside "gettime.h" then every ".c" file that includes it needs recompiling

object files
OBJS = gettime.o lab04.o 

compilation process
gettime.c -> gettime.o lab04.c -> lab04.o 
these .o files are machine code but arent complete programs yet

final executable
EXECS = lab04
eventually Make will create lab04 (the program you actually run)

all: $(EXECS) 
--> all: lab04,
this means: the default thing to build is the program named lab04
so when you type "make" --> Make actually does 'make all' which means build lab04

%.o: %.c $(DEPS)
% --> match any filename, so Make understands this as for any .c file, build a .o file
gettime.c --> gettime.o 
the dependency list is --> %.c $(DEPS)
which expands to %.c gettime.h Makefile
meaning to build something.o you need something.c, gettime.h, and Makefile
if any of those files changes, make recompiles the ".o"

$(CC) $(CFLAGS) -c -o $@ $< == gcc -Wall -g -c -o $@ $<
$@ --> the thing im creating
if were building gettime.o then $@ becomes gettime.o 
$< --> the first dependency
so $< becomes gettime.c 
substituting everything Make executes --> 
gcc -Wall -g -c -o gettime.o gettime.c or gcc -Wall -g -c -o lab04.o lab04.c 
depends on which file its building

lab04: $(OBJS) == lab04: gettime.o lab04.o 
this means: to build the executable lab04, I first need gettime.o and lab04.o

$(CC) $(CLAGS) -o $@ $^ == gcc -Wall -g -o $@ $^
$@ --> target building built, lab04
$^ --> all dependencies, gettime.o lab04.o 
so Make runs gcc -Wall -g -o lab04 gettime.o lab04.o 
links the object files together into one executable

clean:
isnt compiling anything
for convenience

/bin/rm -i *.o $(EXECS) == /bin/rm -i *.o lab04
this removes gettime.o, lab04.o, and lab04 
the -i option means interactive: rm asks for confirmation before deleting each file 
run it by --> "make clean"

all together:

suppose these are your files:
gettime.c 
gettime.h 
lab04.c 
Makefile

first time you run: "make"
make notices nothing has been compiled, so it does:
Compile gettime.c --> gettime.o 
Compile lab04.c --> lab04.o 
Link them together --> lab04

... later, you edit only lab04.c 
when you run: "make"
Make sees: lab04.c changed, gettime.c did not change 
So it does only:
Compile lab04.c --> lab04.o 
Link --> lab04

It doesnt recompile gettime.c, because gettime.o is already up to date.
Thats the main benefit of using a Makefile: it rebuilds only what's necessary based on file dependencies

Calling -->

all you need to type is: "make"
1. You open terminal
2. You change into your project directory:
    cd path/to/your/project
3. Your directory contains something like:
    Makefile
    gettime.c 
    gettime.h 
    lab04.c 
4. You type: "make"
5. "make" reads the Makefile and automatically runs the necessary "gcc" commands
